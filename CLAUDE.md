# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Goal

Create a **patched Qualcomm Firehose programmer** (`fhprg_peek.bin`, msm8909, ARM 32-bit) that can write the aboot partition. The current WP bypass patch (code cave at 0x0803dc8c) does not work — writes still don't persist due to hardware write protection (CMD28 WP groups). The approach now is to **fully decompile** the original binary via Ghidra into readable, correct C so we can understand the write path end-to-end and craft a working patch.

Byte-for-byte reproduction of the original binary is **not a goal**. The original was compiled with ARM Compiler v5 (armcc/RVCT), which we don't have access to. Clang with `-mno-movt` gets literal pool loads right but register allocation and instruction scheduling will never match. Tweaking compiler flags (`-Oz`, `-Os`, `-O1`, `-O2`) does not meaningfully change match rates. The recompilation exists as a verification tool to confirm the decompiled C is functionally correct, not to produce a drop-in replacement.

The `aboot.bin` (Qualcomm LK bootloader) tooling in `tools/preprocess.py` and `src/aboot.ld` is from earlier work on the same infrastructure.

## Workflow

- **Commit regularly** — after each logical change (rename batch, tool fix, build fix, etc.), stage and commit. Don't let changes accumulate across multiple tasks.

## Build Commands

```bash
# fhprg: compile all split modules + link
make fhprg

# fhprg: per-function diff against original
make fhprg-diff

# fhprg: preprocess Ghidra output → src/fhprg.c
python3 tools/preprocess_fhprg.py

# fhprg: split monolithic fhprg.c into per-module files
make fhprg-split

# fhprg: regenerate PROVIDE() symbols + section placements
make fhprg-provides

# aboot: compile decompiled C → object file
make all

# aboot: link compiled code + data blob → ELF at original addresses
make link

# Byte-level + per-function comparison against original
make hybrid-diff

# Run Ghidra headless analysis (requires PyGhidra venv)
make analyze             # fresh analysis
make analyze-reuse       # reuse existing Ghidra project (faster)
```

## Architecture

### Pipeline

1. **Ghidra analysis** (`tools/ghidra_analyze.py` / `ghidra_analyze_fhprg.py`) — Headless decompile of the binary via PyGhidra. Exports per-function C files to `tmp/decompiled/` and function metadata to `tmp/functions.json`.
2. **Preprocessing** (`tools/preprocess_fhprg.py`) — Transforms raw Ghidra C into compilable C: resolves string references, fixes DAT_ globals, handles Ghidra field accessors (`_N_M_` syntax), forward-declares functions, and orders definitions by address.
3. **Splitting** (`tools/split_fhprg.py`) — Splits monolithic `src/fhprg.c` into per-module files in `src/fhprg/` based on original source filenames recovered from assert strings (`DAT_0804cd60` calls containing `"filename.c"` arguments).
4. **Compilation** — Clang cross-compiling to ARM Thumb2: `clang --target=arm-none-eabi -mthumb -march=armv7-a -mfloat-abi=soft -mno-movt -Oz -std=gnu89`. The `-mno-movt` flag forces literal pool loads matching the original binary. Type compatibility header `src/ghidra_types.h` is force-included.
5. **Linking** — Custom linker script (`src/fhprg.ld`) with per-function section placements generated by `tools/gen_fhprg_provides.py`. Data blob provides globals/strings at original addresses.
6. **Verification** — `tools/hybrid_diff.py` does per-function assembly comparison with multi-level normalization (mnemonic, register, literal pool, branch target).

### Key Files

- `src/ghidra_types.h` — Type compatibility header mapping all Ghidra-emitted types (undefined4, code*, CONCAT macros, SUB/SEXT/ZEXT macros) and ARM intrinsic stubs (CP15 ops, cache maintenance, barriers, SMC) to real implementations.
- `src/fhprg/` — Split source modules (~93 files), named after original Qualcomm source files where identifiable. Each includes `globals.h`.
- `src/fhprg/globals.h` — Forward declarations for all functions and extern globals.
- `src/fhprg.c` — **Generated file** (do not edit directly). Produced by `preprocess_fhprg.py` from Ghidra output, then split by `split_fhprg.py`.
- `tmp/fhprg_syms.txt` — Function address/type/name index (addr T name per line).
- `tmp/fhprg_renames.json` — Function rename map (57 functions identified from logging strings, dispatch tables, and DAL attach calls).
- `tmp/fhprg_provides.ld` — PROVIDE() directives for ~1262 external symbols (DAT_, PTR_, etc.).
- `tmp/fhprg_sections.ld` — Per-function section placements (1878/1880 functions placed at original addresses).
- `tmp/decompiled/` — Individual per-function `.c` files from Ghidra.

### Compiler Settings

fhprg uses Clang 19 cross-compiling for ARMv7-A Thumb2 mode with soft float. The `-std=gnu89` flag matches K&R-style function declarations Ghidra emits. `-Wno-everything` suppresses all warnings since Ghidra output is inherently messy. GCC does not support `-mno-movt` for armv7-a, making Clang the required choice.

aboot uses `arm-none-eabi-gcc` compiling for ARMv7-A ARM mode with soft float.

### Decompilation Quality

The focus is on producing **readable, correct C** — not on matching compiler output. Key techniques:
- Recover original source filenames from assert/error handler strings
- Rename functions using logging call arguments (boot_log_message, firehose_log, dal_device_attach)
- Identify the firehose command dispatch table to name handler functions
- Fix Ghidra artifacts: field accessors, type casts, comment-merged definitions
- The preprocessor scripts (`preprocess_fhprg.py`, `split_fhprg.py`) are the main levers for improving decompilation quality
